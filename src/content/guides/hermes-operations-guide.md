# Hermes Agent 운영 가이드: 프로필·게이트웨이·cron·Kanban 실전 운영법

Hermes를 안정적으로 운영하는 핵심은 모델을 자주 바꾸는 데 있지 않다. 프로필을 책임 단위로 나누고, 게이트웨이는 운영체제가 복구하게 두며, 정기 실행은 얇은 cron과 추적 가능한 Kanban 카드로 분리하고, 모든 변경의 근거를 문서와 검증 결과로 남기는 데 있다. 이 글은 Hermes Agent v0.18.2를 기준으로 여러 프로필, 반복 작업, 리뷰가 함께 돌아가는 환경에서 실제로 썼던 운영 규칙을 공개 가능한 형태로 정리한다. 가장 먼저 할 일은 `hermes status`, `hermes profile list`, `hermes gateway status`, `hermes cron status`를 차례로 실행해 현재 상태를 관찰하는 것이다. 상태를 모른 채 재시작하거나 설정을 덮어쓰는 행동이 가장 비싼 장애를 만든다.

## 빠르게 훑어보기

- 프로필은 말투가 아니라 메모리·설정·cron·권한 경계다. 역할과 위험이 다르면 분리한다.
- 게이트웨이 장애의 첫 대응은 상태와 로그 확인이다. launchd가 이미 복구하는지 확인하기 전에는 재시작하지 않는다.
- cron은 “언제 실행할지”만 맡긴다. 재시도, 의존성, 리뷰, 산출물 감사가 필요하면 Kanban 카드로 넘긴다.
- Kanban의 `blocked`는 사람의 판단이 없으면 진행할 수 없는 경우에만 쓴다. 산출물이 완성되어 리뷰만 남았다면 build 카드는 완료 처리해 다음 카드를 열어야 한다.
- 공개 문서와 로그에는 계정·내부 주소·식별자·인증 문자열을 넣지 않는다. 예시는 `<PROFILE_NAME>`, `<SERVICE_LABEL>`, `<CREDENTIAL_VALUE>`처럼 치환한다.

## 이 가이드가 다루는 운영 범위

Hermes는 터미널, 메신저, cron, 도구 호출, 메모리, 스킬과 여러 프로필을 하나의 런타임으로 묶는다. 기능이 많다는 이유만으로 모두를 하나의 대화형 프로필에 넣으면 실패의 원인과 권한의 경계가 사라진다. 반대로 역할을 지나치게 잘게 나누면 상태가 흩어지고, 누가 무엇을 고칠 수 있는지 알 수 없게 된다. 실전에서 유효했던 균형은 “책임이 다르거나 접근 가능한 데이터가 달라질 때만 프로필을 나눈다”이다.

예를 들어 오케스트레이션은 작업을 분해하고 의존성을 관리하며 사람의 결정을 모은다. 운영·보안은 시스템 상태와 증거를 다룬다. 독립 리뷰는 작성자와 다른 관점에서 결과를 검증한다. 연구·콘텐츠는 공개 소스와 발행 흐름을 맡는다. 각 역할은 필요한 도구와 저장 위치가 다르다. 이 분리는 조직도를 흉내 내기 위한 장식이 아니라, 잘못된 자동화가 다른 역할의 상태를 바꾸지 못하게 하는 안전장치다.

현재 설치 상태를 읽는 기본 명령은 다음과 같다.

```bash
hermes --version
hermes status
hermes profile list
hermes gateway status
hermes cron status
```

명령의 출력에서 먼저 확인할 것은 “정상”이라는 한 단어가 아니다. 어떤 프로필의 서비스인지, 실행 중인지, 최근 오류가 있는지, cron 스케줄러가 살아 있는지다. 이상이 보이면 해당 프로필의 공개 가능한 로그 요약과 마지막 변경 시점을 같이 기록한다. 원인 모르는 재시작을 반복하면 원래 오류와 재시작으로 생긴 오류를 구분할 수 없게 된다.

## 프로필은 어떻게 설계해야 하나

프로필 하나는 독립된 운영 단위다. 프로필은 모델 설정만 따로 갖는 것이 아니라, 그 역할의 메모리·스킬·cron·게이트웨이 연결과 작업 습관을 함께 가진다. 따라서 “새 캐릭터가 필요해서”보다 아래 질문의 답이 달라질 때 만드는 편이 좋다.

1. 이 역할이 읽어도 되는 데이터 범위는 무엇인가?
2. 이 역할이 외부로 발신해도 되는 채널은 어디인가?
3. 실패를 누가 검토하고 어떤 증거로 판단하는가?
4. 반복 작업이 다른 역할의 일정이나 권한과 충돌하는가?
5. 장기 메모리에 남아도 되는 지식의 성격이 같은가?

첫 배치는 보통 세 종류면 충분하다. 조율자 프로필은 Kanban과 일정, 문서 handoff를 관리한다. 운영 프로필은 상태 점검과 검증 증거를 담당한다. 생산 또는 연구 프로필은 실제 산출물을 만든다. 독립 검토가 필요해지면 네 번째로 리뷰 프로필을 둔다. 처음부터 역할마다 프로필을 만들지 말고, 실제 충돌이나 권한 과잉이 보일 때 분리한다.

프로필을 확인하고 새로 만들 때는 공식 CLI를 사용한다.

```bash
hermes profile list
hermes profile show <PROFILE_NAME>
hermes profile create <NEW_PROFILE_NAME>
```

프로필을 만든 직후에는 곧바로 외부 채널이나 반복 실행을 붙이지 않는다. 먼저 역할 문서에 목적, 금지 행위, 허용된 데이터 범위, 실패 시 handoff 위치를 적고, 단발성 작업으로 도구·출력 경계를 확인한다. 이 순서를 지키면 “동작은 하지만 잘못된 곳에 기록하는” 초기 설정 오류를 줄일 수 있다.

### 로스터의 단위는 사람 이름이 아니라 책임이다

로스터 문서는 활성 프로필, 각 프로필의 권한, 은퇴한 역할, handoff 원칙을 짧게 기록해야 한다. 중요한 것은 개인적 별명이 아니라 책임의 유일성이다. 가령 게이트웨이 서비스 감독을 한 역할에, 팀 작업 조율을 다른 역할에, 독립 리뷰를 세 번째 역할에 할당했다면 같은 조치를 여러 프로필이 동시에 실행하지 않도록 해야 한다.

이 원칙이 필요한 이유는 중복 복구 사고 때문이다. 하나의 게이트웨이를 여러 감시자가 동시에 재시작하면, 정상 복구 직후 다른 감시자가 오래된 상태를 보고 다시 bootstrap할 수 있다. 그 결과 서비스 정의의 상태가 흔들리고, 실제 원인을 가리는 재시작 로그만 늘어난다. 책임을 한 곳에 두고 다른 프로필은 관찰과 문서 보고만 하게 하면 이 경쟁 조건을 줄일 수 있다.

## 게이트웨이는 어떻게 운영해야 하나

게이트웨이는 사용자의 메시지와 에이전트 실행을 잇는 입구다. 그래서 “프로세스가 살아 있다”는 것만으로 정상이라 할 수 없다. 올바른 프로필과 서비스가 실행 중인지, 필요한 플랫폼만 연결했는지, 재시작 정책이 한 곳에만 있는지를 함께 봐야 한다.

macOS에서는 launchd가 첫 번째 복구 계층이다. 서비스가 비정상 종료했을 때 `KeepAlive` 설정이 있다면 운영자가 개입하지 않아도 프로세스를 되살릴 수 있다. 따라서 장애 대응은 다음 순서를 권장한다.

```bash
hermes gateway status
launchctl list | grep '<SERVICE_LABEL>'
# 최근 오류는 개인 식별 정보가 없는 범위에서 확인한다.
grep -i 'error\|failed' <PROFILE_HOME>/logs/gateway.log | tail -20
```

첫 번째 단계는 Hermes가 보는 상태, 두 번째 단계는 OS가 보는 서비스 상태, 세 번째 단계는 최근 실패 신호를 비교한다. 이 세 증거가 서로 맞지 않을 때만 더 깊게 조사한다. 서비스가 이미 정상 PID로 복구됐다면 수동 재시작은 하지 않는다. 재시작은 진단 절차가 아니라 상태를 바꾸는 조치다.

### 단일 책임 재시작 정책

안전한 운영 모델은 세 계층으로 나뉜다.

| 계층 | 질문 | 담당 행동 |
| --- | --- | --- |
| L1 | 프로세스가 죽었는가 | launchd가 자동 복구한다 |
| L2 | 서비스는 살아 있지만 health가 반복 실패하는가 | 지정된 watchdog이 증거를 수집하고 정책에 따라 조치한다 |
| L3 | 메모리 누수, 코드 반영, 특수 장애인가 | 명시적으로 권한을 받은 운영자가 원인별 절차를 실행한다 |

핵심은 L1의 자동 복구가 있는 상태에서 모든 역할이 L2와 L3을 흉내 내지 않는 것이다. 특히 콘텐츠 작성자나 일반 작업자는 자기 게이트웨이를 재시작하면 안 된다. 연결 이상을 발견하면 시간, 관찰한 상태, 이미 확인한 명령 결과를 문서 handoff로 남긴다. 운영 담당자는 그 증거를 바탕으로 정책 범위 안에서 판단한다.

공식 CLI로 서비스 제어가 필요한 권한을 가진 경우에도 순서는 같다.

```bash
hermes gateway status
# 정책상 허용되고 원인이 확인된 경우에만 사용한다.
hermes gateway restart
hermes gateway status
```

재시작 전후의 상태를 같은 명령으로 비교해야 한다. “명령이 성공했다”와 “사용자가 다시 연결됐다”는 다른 사실이다. 후자의 증거는 gateway 상태, 연결 로그, 해당 플랫폼의 허용된 health 신호로 따로 확인한다.

### launchd에서 자주 놓치는 문제

launchd는 대화형 셸과 환경이 다르다. 터미널에서는 찾던 실행 파일이 서비스에서는 찾지 못하는 경우가 흔하다. 서비스용 스크립트는 암묵적인 셸 초기화에 기대지 말고 필요한 실행 경로를 명시하거나 절대 경로를 사용한다. 또 하나의 실패 패턴은 같은 캐시나 플러그인 디렉터리를 두 프로세스가 동시에 갱신하는 것이다. 설치·업데이트를 하는 보조 프로세스와 gateway를 동시에 기동하면 파일 이름 변경 경쟁으로 반복 종료가 일어날 수 있다. 이때 해결은 재시작 횟수를 늘리는 것이 아니라, 동시 설치 경로를 제거하고 하나의 소유자만 갱신하게 만드는 것이다.

운영 기록에는 실제 장비 주소나 계정명을 넣지 말고 다음처럼 재현 가능한 범주만 남긴다.

```text
관찰: gateway 연결이 끊김
확인: launchd 서비스는 실행 중, 최근 로그에 플러그인 초기화 실패
조치: 자동 복구 여부를 관찰하고 동시 설치 작업을 중지
판정: 서비스 재기동 후 상태 명령과 연결 로그로 복구 확인
```

## cron은 얇게, 작업의 수명은 Kanban으로

cron은 반복의 시계다. 매일 보고서를 만들거나 정해진 시간에 새 작업을 시작하는 일에 적합하다. 하지만 retry, 의존성, 코드 리뷰, 빌드 검증, 사람 승인처럼 여러 상태가 필요한 일을 cron 하나에 전부 넣으면 실패가 숨는다. 실전 규칙은 간단하다. cron은 “언제 시작할지”를 맡고, Kanban은 “무엇을 어떤 증거로 끝낼지”를 맡는다.

다음 세 질문 중 하나라도 예라면 카드로 넘긴다.

1. 실패하면 재시도 또는 사람이 보는 리뷰가 필요한가?
2. 다른 작업이 끝나야 시작하거나 다른 역할과 조율해야 하는가?
3. 조용히 실패했을 때 비용이 큰가?

전부 아니라면 짧은 점검, 멱등적인 감시, 변화가 있을 때만 알리는 작업은 cron에 남아도 된다. 예를 들어 상태 파일을 읽고 차이가 있을 때만 출력하는 watcher는 사람이 개입할 필요가 없는 순수 반복 작업이다. 반면 발행 글 생성, PR 작성, 배포 전 검증은 카드의 상태·코멘트·의존성이 필요하다.

현재 작업 목록과 스케줄러는 다음 명령으로 본다.

```bash
hermes cron list
hermes cron status
hermes cron run <JOB_ID>
```

새 LLM 기반 cron은 provider와 model을 명시적으로 고정하는 편이 안전하다. 전역 모델을 바꿨을 때 의도치 않게 다른 품질·비용·권한 조건이 적용되는 것을 막기 위해서다. 반대로 단순 스크립트 감시에는 agent 실행이 필요하지 않을 수 있다. 출력이 비어 있으면 조용히 끝나고, 변화가 있을 때만 짧은 메시지를 남기도록 설계하면 잡음과 비용을 함께 줄일 수 있다.

### cron 이름과 산출물 계약

작업 이름에는 소유 프로필, 업무 영역, 행동을 넣는다. 예를 들어 `<PROFILE>-<AREA>-<ACTION>`처럼 읽는 사람이 책임을 추정할 수 있어야 한다. 이름을 바꾸는 일은 생각보다 위험하다. 문서, 스크립트, 모니터링, seeder가 기존 이름을 참조할 수 있으므로 새 이름을 추가하고 참조를 옮긴 뒤 유예 기간 후 이전 이름을 없애는 방식이 낫다.

더 중요한 것은 성공의 정의다. “완료”라는 문장만 남기는 cron은 실제 파일을 만들지 않았어도 정상으로 보일 수 있다. 산출물이 있는 작업은 완료 기록에 파일 유형, 개수, 크기 또는 신선도를 남긴다. 주기 감사는 기대하는 파일 패턴과 최소 크기, 마지막 갱신 시점을 검사한다. 이것이 이른바 좀비 작업을 막는 방법이다. 프로세스는 살아 있지만 며칠째 같은 실패를 반복하거나 빈 결과만 내는 작업을 상태 코드만으로는 발견하기 어렵다.

```text
STATUS=OK
ARTIFACT=<PUBLIC_OR_REDACTED_PATH>
FRESHNESS=<UTC_TIMESTAMP>
COUNT=<NUMBER>
NEXT=<NEXT_ACTION_OR_NO_OP>
```

이 형식은 민감한 내부 경로를 공개하라는 뜻이 아니다. 내부 로그에서는 권한 정책에 맞는 실제 경로를 쓰되, 사람과 다른 프로필이 판단해야 하는 handoff에는 공개 가능한 식별자나 치환값을 사용한다.

### 중복 실행과 멱등성

같은 tick이 두 번 돌거나 이전 실행이 늦게 끝나는 상황은 항상 고려해야 한다. 발행, 알림, 외부 쓰기처럼 중복 부작용이 있는 작업은 idempotency key 없이 무한 재시도하면 안 된다. 날짜나 입력 버전을 포함한 키를 두고, 이미 만든 결과를 다시 만들지 않는지 확인한다. Kanban을 seeder로 쓰는 경우에는 같은 날짜·같은 업무가 한 장의 카드로 수렴해야 한다.

```bash
# 개념 예시: 실제 환경의 board와 assignee를 확인한 뒤 사용한다.
hermes kanban create "일일 검증" --assignee <PROFILE_NAME> \
  --idempotency-key "daily-verification-$(date -u +%F)"
```

이 명령은 조직마다 옵션과 권한이 다를 수 있다. 중요한 것은 실행 전 `hermes profile list`로 실제 할당 대상을 확인하고, 동일 업무의 이전 카드가 있는지 확인하는 것이다. 존재하지 않는 담당 이름은 조용히 ready 상태에 남는 실패로 이어질 수 있다.

## Kanban을 작업 이력으로 쓰는 법

Kanban은 체크리스트 앱보다 엄격한 작업 수명 모델이다. 카드에는 목표, 수용 기준, 담당자, 부모 작업, 코멘트, 실행 이력이 남는다. cron이 정기적으로 카드를 심고 dispatcher가 준비된 카드를 실행하며, 작업자는 결과와 증거를 남긴다. 이 구조의 가치는 “누가 바쁜가”가 아니라 “실패가 어디에서 멈췄고 다음 행동은 무엇인가”를 알 수 있다는 점이다.

작업자는 시작할 때 카드 전문과 부모 handoff를 읽는다. 그 뒤 실제 작업 공간에서 산출물을 만들고, 긴 실행이면 heartbeat를 남긴다. 결과가 완성됐다면 요약과 검증 명령, 변경 파일, 남은 위험을 구조화해 완료한다. 작업이 사람의 사업 판단이나 권한 승인을 기다릴 때만 block한다.

```bash
hermes kanban show <TASK_ID>
hermes kanban list --mine
```

에이전트 worker는 보통 전용 도구로 상태를 바꾸지만, 사람이 점검할 때는 CLI 조회가 유용하다. 중요한 점은 build 카드와 review 카드의 역할을 섞지 않는 것이다. build가 PR과 테스트 증거를 만들었다면 “리뷰를 기다린다”는 이유만으로 build를 blocked로 두지 않는다. 부모가 done이어야 자식 리뷰 카드가 준비 상태가 되는 설계라면, build를 완료하고 리뷰 카드를 다음 단계로 넘긴다. block은 원저자나 사람이 없으면 논리적으로 진행할 수 없는 경우다.

### 실전에서 반복된 Kanban 함정 네 가지

**첫째, 완료한 build를 review 대기라며 blocked로 두는 함정이다.** 이 상태는 후속 카드의 의존성을 막는다. 산출물과 테스트가 준비됐으면 완료 기록을 남기고, 리뷰의 승인 여부는 별도 카드에서 다룬다.

**둘째, 존재하지 않는 프로필에 배정하는 함정이다.** dispatcher는 실재하지 않는 담당자를 자동으로 고쳐 주지 않는다. 먼저 `hermes profile list`로 이름을 확인하고, 작업 설명에도 실제 역할과 책임을 적는다.

**셋째, 열린 PR을 참조하는 리뷰 카드가 재실행 방지 장치에 걸리는 함정이다.** 중복 PR을 막는 보호가 리뷰 카드에도 적용되면 ready인데 실행되지 않는 상황이 생길 수 있다. 이때는 동시성 부족이라고 단정하지 말고 카드 이벤트와 재실행 방지 사유를 확인한다. 사람의 최종 머지가 필요한 흐름이라면 검증 결과를 남겨 안전하게 보류하는 것이 자동 재시작보다 낫다.

**넷째, 노출되는 상태만 보고 작업이 끝났다고 믿는 함정이다.** 카드의 최근 요약, 코멘트의 산출물, 실제 브랜치 또는 빌드 결과를 함께 봐야 한다. 상태 하나만으로는 완료 증거가 되지 않는다.

### 코멘트는 다음 사람이 실행할 수 있게 쓴다

좋은 handoff는 “완료했습니다”가 아니다. 무엇을 했고, 어떤 명령을 어떤 결과로 실행했으며, 다음 사람이 무엇을 판단하면 되는지 3단으로 적는다.

```text
Signal: <관찰한 변화 또는 실패>
So What: <서비스·사용자·후속 작업에 주는 의미>
Action: <다음 담당자가 실행하거나 판단할 한 가지>
Evidence: <검증 명령과 exit code, 공개 가능한 산출물 식별자>
```

이 구조는 연구 보고에도, 빌드 handoff에도 쓸 수 있다. 특히 오류가 난 경우에는 로그 전체를 붙이는 대신 실패한 명령, 오류 범주, 이미 시도한 안전한 조치를 적는다. 다음 사람이 같은 재시작이나 같은 검색을 반복하지 않게 하는 것이 목적이다.

## 시크릿·개인정보·공개 문서의 경계

운영 가이드는 현실적인 명령과 사례가 있어야 하지만, 현실의 식별자를 노출하면 안 된다. 공개 대상 Markdown, 스크린샷, 빌드 로그에는 다음을 넣지 않는다.

- 인증 문자열, API credential, 세션 값, 복구 코드
- 계정명, 개인 전화번호, 메일 주소, 개인 장비 경로
- 사설 네트워크 주소, 채널·대화 식별자, 실서비스 접근 주소
- 환경 파일의 실제 내용과 자격 증명 파일 경로

예시는 항상 `<CREDENTIAL_VALUE>`, `<PROFILE_HOME>`, `<SERVICE_LABEL>`, `<CHAT_IDENTIFIER>` 같은 치환값으로 쓴다. 치환값이 있다고 해서 검증을 생략해선 안 된다. 발행 직전에는 콘텐츠 파일과 생성된 결과물을 대상으로 패턴 검사를 실행하고, 결과가 비어 있는지 확인한다. 아래 예시는 탐지용 규칙이며 실제 값은 출력하지 않는다.

```bash
# 공개 가이드 대상에서 민감한 영문 표식과 사설 주소 형태를 찾는다.
p1='1''00\.'; p2='4''9\.'
grep -nEi "credential-marker|${p1}|${p2}" \
  src/content/guides/hermes-operations-guide.md
```

이 가이드 본문에는 위 명령을 설명하기 위해 탐지 단어가 등장한다. 실제 발행 파일을 검사할 때는 운영 문서의 검사 명령과 콘텐츠의 설명 문구를 분리하거나, 검사 대상에서 설명 자체를 제외하는 방법을 선택해야 한다. 더 좋은 방법은 CI에서 허용된 플레이스홀더만 통과시키고 실제 credential 패턴과 주소만 차단하는 별도 검사 규칙을 만드는 것이다. 단순 문자열 검색은 교육용 문장까지 잡을 수 있으므로, 최종 판정은 정규식과 사람이 함께 한다.

자격 증명은 설정 파일과 섞지 않는다. 장기 동작 설정은 검토 가능한 설정 파일에, 비밀값은 전용 credential 저장소 또는 환경 주입 경로에 둔다. 중요한 것은 저장 위치를 공개하는 것이 아니라, 어떤 역할이 어떤 비밀값에 접근할 수 있는지를 최소화하는 것이다. 연구·콘텐츠 프로필이 운영 credential을 읽을 이유가 없다면 그 권한을 주지 않는다.

## 운영 점검 루틴

매일 모든 서비스를 손으로 확인할 필요는 없다. 대신 실패의 비용과 변화 빈도에 따라 점검 주기를 다르게 둔다. 다음은 작은 팀에 맞는 보수적 루틴이다.

| 주기 | 확인 대상 | 확인 질문 |
| --- | --- | --- |
| 매 실행 | 작업 카드와 변경 범위 | 이 작업의 목표·권한·검증 명령이 명확한가 |
| 매일 | gateway·cron 요약 | 죽었거나 반복 실패한 서비스가 있는가 |
| 매주 | 산출물 신선도·cron 오류 | 성공 보고만 하고 결과를 만들지 않은 잡이 있는가 |
| 매월 | 프로필·권한·스킬 | 더 이상 필요 없는 접근 권한과 중복 책임이 있는가 |

일일 점검은 짧아야 지속된다.

```bash
hermes status
hermes gateway status
hermes cron status
hermes kanban stats
```

이 결과를 보고 즉시 설정을 바꾸기보다, 이상 징후를 분류한다. 첫째 실제 장애인지, 둘째 자동 복구 중인지, 셋째 오래된 상태 표시인지, 넷째 사람이 판단해야 하는 변화인지 나눈다. 이 분류가 없으면 건강 점검 자동화가 그 자체로 재시작 자동화가 된다.

주간 점검에서는 산출물 계약을 본다. 최근 7일 동안 오류 또는 미실행 상태가 계속된 잡, 기대 파일이 갱신되지 않은 보고서, 완료 기록은 있으나 검증 증거가 없는 카드가 후보가 된다. 이때도 해결책을 한꺼번에 적용하지 않는다. 오류 원인을 작업 정의, 권한, 네트워크, 외부 의존성, 잘못된 가정으로 분류해 작은 카드로 만든다.

## 장애를 조사하는 순서

운영 장애는 보통 세 부류로 나뉜다. 서비스가 실제로 죽은 경우, 서비스는 살아 있지만 외부 연결이 안 되는 경우, 작업은 실행되지만 산출물이 없는 경우다. 각 부류에 같은 재시작을 쓰면 안 된다.

### 서비스가 죽은 것처럼 보일 때

1. `hermes gateway status`로 Hermes가 보는 상태를 확인한다.
2. `launchctl list`로 서비스가 OS 수준에서 살아 있는지 본다.
3. 최근 로그에서 종료·초기화·권한 오류 중 어느 범주인지 찾는다.
4. launchd가 자동 복구한 상태라면 관찰을 계속하고 수동 재시작하지 않는다.
5. 명시된 담당자와 정책이 있을 때만 원인에 맞는 재기동을 고려한다.

### cron이 실행됐지만 결과가 없을 때

1. 스케줄러와 해당 잡의 마지막 상태를 분리해서 확인한다.
2. stdout만 보고 성공으로 보지 말고 기대 산출물의 존재·크기·갱신 시점을 확인한다.
3. 입력이 비어도 정상인 watcher인지, 실제로 만들어야 할 결과가 누락된 것인지 구분한다.
4. 멱등 키, 저장 경로 권한, 모델 고정값, 외부 의존성 변화를 점검한다.
5. 재시도가 중복 발행을 만들 수 있으면 먼저 side effect를 막고 카드로 전환한다.

### Kanban이 ready인데 실행되지 않을 때

1. 부모 카드가 실제로 done인지 확인한다.
2. 담당 프로필이 실제 존재하는지 확인한다.
3. 최근 이벤트에서 재실행 방지나 active PR 관련 사유가 있는지 본다.
4. 단순 동시성 부족으로 추측하지 말고 이벤트·최근 실행·현재 작업 수를 함께 본다.
5. 사람의 병합이나 사업 판단이 필요한 경우에는 증거를 남기고 block한다.

## FAQ

### Hermes 운영을 시작할 때 무엇부터 확인하나

먼저 버전, 런타임, 프로필, gateway, cron을 각각 확인한다. `hermes --version && hermes status && hermes profile list && hermes gateway status && hermes cron status`를 실행하면 현재 상태를 한 번에 훑을 수 있다. 오류가 보이면 다음 명령을 실행하기 전에 어떤 프로필과 어떤 구성요소에서 나온 신호인지 기록한다.

### 프로필은 언제 분리해야 하나

역할의 데이터 접근, 외부 발신 권한, 실패 책임, 장기 메모리 범위 중 하나라도 다르면 분리를 검토한다. 먼저 `hermes profile list`로 현재 프로필을 보고, 새 역할의 책임 문서를 만든 다음 `hermes profile create <NEW_PROFILE_NAME>`로 최소한의 프로필을 만든다. 단지 말투가 다르다는 이유만으로 만들 필요는 없다.

### gateway가 응답하지 않으면 바로 재시작해도 되나

아니다. 먼저 `hermes gateway status`와 `launchctl list | grep '<SERVICE_LABEL>'`로 자동 복구 여부를 확인한다. 이미 정상 PID가 있으면 재시작하지 말고 최근 오류 범주를 조사한다. 재시작 권한이 명시된 담당자라면 원인 확인 후 `hermes gateway restart`를 실행하고 다시 상태를 확인한다.

### cron과 Kanban은 어떻게 나누나

정기 시각에 시작하는 일은 cron에 둔다. 재시도, 의존성, 리뷰, 산출물 검증, 사람 승인이 하나라도 필요한 일은 Kanban 카드로 넘긴다. 확인은 `hermes cron list`와 `hermes kanban show <TASK_ID>`로 시작한다. cron은 얇은 seeder가 되고, 카드가 실행 이력과 증거를 가진다.

### cron 작업 이름은 어떻게 붙이나

`<PROFILE>-<AREA>-<ACTION>`처럼 소유자·업무 영역·행동이 보이게 한다. 이름을 바꾸기 전에는 기존 참조를 조사하고 새 이름을 병행한 뒤 옮긴다. `hermes cron list`로 현재 이름을 확인하고, 변경 후에는 `hermes cron status`로 스케줄러와 마지막 실행을 재확인한다.

### Kanban 카드가 리뷰만 남았을 때 blocked로 두나

아니다. build 산출물과 테스트가 준비됐다면 build 카드는 완료 처리하고 후속 리뷰 카드가 열리게 한다. `blocked`는 원저자나 사람의 결정 없이는 논리적으로 진행할 수 없는 경우에 쓴다. 상태를 바꾸기 전 `hermes kanban show <TASK_ID>`로 부모·자식·최근 실행 요약을 확인한다.

### 긴 작업의 heartbeat는 언제 남기나

수 분 이상 걸리는 빌드, 대량 검사, 크롤링, 변환 작업에는 진행 증거를 주기적으로 남긴다. heartbeat에는 “작업 중” 대신 처리량이나 단계처럼 판단 가능한 정보를 쓴다. 예를 들어 `검증 42/100 완료, 실패 0`처럼 적으면 stale 실행과 정상 장기 실행을 구분할 수 있다.

### 공개 가이드에서 무엇을 치환해야 하나

계정, 개인 경로, 사설 주소, 채널 식별자, 인증 문자열을 전부 치환한다. 예시는 `<PROFILE_HOME>`이나 `<CREDENTIAL_VALUE>`를 사용한다. 발행 전에는 콘텐츠 대상으로 민감 패턴 검사를 하고, 생성된 HTML에도 같은 정보가 남지 않았는지 확인한다.

### 설정을 바꾼 뒤 어떤 검증을 하나

설정 형식을 확인한 뒤 실제 명령으로 효과를 본다. gateway 변경이면 `hermes gateway status`, cron 변경이면 `hermes cron status`와 필요 시 `hermes cron run <JOB_ID>`, 프로필 변경이면 `hermes profile show <PROFILE_NAME>`을 실행한다. 파일만 수정하고 서비스 상태를 추측하지 않는다.

### 작업이 끝났다는 증거는 무엇인가

사람의 서술이 아니라 산출물과 검증 결과다. 변경 작업이라면 변경 파일, 실행한 검사 명령, exit code, PR 또는 검토 위치를 남긴다. 자동 작업이라면 결과 파일의 신선도·개수·크기 같은 계약을 남긴다. Kanban에서는 `hermes kanban show <TASK_ID>`로 최근 요약과 코멘트를 함께 확인한다.
